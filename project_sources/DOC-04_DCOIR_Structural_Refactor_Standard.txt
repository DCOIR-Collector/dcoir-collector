DCOIR Structural Refactor Standard

Purpose
- This document defines the current governed standard for structural refactors of growth-prone readable files in the AFRICOM_SOC_IR / DCOIR project.
- Use this standard when a governed file, log, todo structure, workflow artifact, or other readable project source is likely to become brittle if it remains monolithic.

Scope
- This standard applies to governed readable text in the GitHub-primary repo working line, especially under project_sources/ and knowledge/.
- This standard complements the current setup guide and layout spec. It does not replace Project Instructions, CP-01, or CP-02.
- This standard does not apply to retained binary assets in supporting_assets/ except where those assets require refreshed references after a structural change.

Split-trigger rules
- Split a governed file when its size, change frequency, or mixed-purpose structure makes safe maintenance harder than a bounded split would.
- Prefer splitting when a file combines multiple durable roles that future sessions are likely to update independently.
- Prefer splitting when future resume quality, navigation, or operator guidance would improve from clearer child-file boundaries.
- Do not split merely because a file is long. Split when the role boundaries are clear and the resulting structure will reduce future maintenance risk.

Canonical split pattern
- Prefer an index-plus-child-file model for growth-prone governed structures.
- Keep one stable parent, index, wrapper, or summary file when future sessions need a predictable entrypoint.
- Place durable subtopics, lanes, or role-bounded content into child files with explicit names.
- Treat the stable parent or wrapper as the first navigation point, and the child files as the detailed working surface.
- For executable or runtime-sensitive structures, preserve the canonical runtime filename and move readable source detail into role-bounded child files.

Naming rules
- Preserve stable runtime or entry filenames when operators, skills, or docs already depend on them.
- Use role-based names, not arbitrary size-based names.
- Prefer numbered child files only when the sequence itself is meaningful and stable.
- Prefer names that communicate scope, lane, or role clearly enough that a future session can choose the correct file without hidden context.
- Do not create parallel naming models for the same structure without a documented reason.

Functional-boundary rules
- Define split boundaries by role, workflow lane, or durable topic, not by arbitrary line count.
- Group content that is usually validated, updated, or resumed together.
- Separate content that is likely to evolve independently.
- Preserve one obvious first-entry file for resume and governance whenever the structure is split.

Dependency-first cutover sequence
- Define the target structure first.
- Create the new child files before changing any stable entrypoint, wrapper, index, or parent file that points to them.
- Verify every new dependency exists and is readable before cutover.
- Cut over the stable parent, wrapper, or index only after the child files exist.
- Refresh control-plane, workflow, and continuity references immediately after the cutover so authority does not drift.

Required verification
- Verify every newly created child file exists at the intended governed path.
- Verify the stable parent, wrapper, or index points only to live current files.
- Verify CP-01, CP-02, relevant evergreen docs, todo posture, and handoff posture stay aligned when the refactor changes the live working model.
- Verify downstream repo delivery artifacts do not silently describe the old structure.
- Treat commit success as insufficient; always verify readback of the final repo state.

Traceability requirements
- Preserve cross-file traceability whenever a governed file is split.
- Ensure the parent, index, or wrapper tells future sessions where the child files live.
- Keep enough visible context in parent or index files that resume quality remains clean.
- When a split changes the project working model, refresh the handoff and todo posture so future sessions do not infer the old layout.

Legacy-artifact handling
- Review legacy or orphaned split-attempt artifacts before cleanup.
- Remove nothing until dependency checks confirm nothing still relies on those artifacts.
- Prefer content comparison over filename comparison when deciding whether a legacy artifact is a real duplicate or still carries unique information.

Control-plane refresh set
- Refresh CP-01 when the current governed working set or current next work item changes materially.
- Refresh CP-02 whenever the project should remember why the structural change happened and what was aligned with it.
- Refresh DOC-01 and DOC-03 when the stable workflow or layout model changes.
- Refresh LOG-01, LOG-03, and the split todo files when the active work line, queued follow-ons, or continuity posture changes.

Current active structural-standard pass

- Inventory remaining large governed logs, continuity files, and other likely split candidates.
- Normalize naming and folder conventions where older experimental paths remain.
- Review remaining legacy or orphaned split-attempt artifacts before any cleanup.
- Check whether repo delivery artifacts still lag the current split todo and task-memory posture.
- After the standard is in place and the candidate inventory is clearer, resume the extracted-folder integration pass under the new structural rules.
