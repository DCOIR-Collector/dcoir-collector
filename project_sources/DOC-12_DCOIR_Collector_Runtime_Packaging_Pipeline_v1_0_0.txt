DCOIR Collector Runtime Packaging Pipeline v1.1.0

Purpose
- Define a validate-and-build workflow for packaging the current collector wrapper, maintained collector part set, and PowerShell harness into a versioned delivery zip.
- Preserve the collector line as a governed, reviewable, and transport-safe package rather than a partial wrapper-only archive.
- Keep the collector packaging line aligned to the operator’s current mail-filter constraint and the existing compiled-runtime lane from maintained source parts.

Current source model
- `project_sources/DCOIR_Collector.ps1` is the current readable collector wrapper source.
- `project_sources/collector_parts/*.ps1` is the required maintained collector part set.
- `project_sources/run_DCOIR_Tests.ps1` is the current readable PowerShell harness source.
- `project_sources/Collector_Runtime_Package_Manifest.json.txt` defines the compile and delivery package contract.
- `supporting_assets/DCOIR_Collector.zip` remains the retained supporting runtime asset and should not be overwritten directly from the transport-safe delivery package without restoring native runtime filenames first.

Delivery rules
- The workflow artifact is a transport-safe delivery package.
- The collector wrapper and collector part set are not shipped as separate runtime dependencies in the delivery package.
- The workflow must first compile a single runtime `DCOIR_Collector.ps1` from the maintained source parts.
- The delivery package then ships the compiled collector runtime and the harness as `.txt`-suffixed script files.
- Before local runtime use, strip only the terminal `.txt` from script entries after extraction.

Workflow
1. Confirm the current control plane.
2. Edit `project_sources/DCOIR_Collector.ps1`, `project_sources/collector_parts/*.ps1`, and or `project_sources/run_DCOIR_Tests.ps1` as needed.
3. Update `project_sources/Collector_Runtime_Package_Manifest.json.txt` if package contents or delivery-safe filenames changed.
4. Run `build_dcoir_collector_runtime_package.py`.
5. Review the workflow summary and uploaded artifacts.
6. Treat the output as a delivery package unless a later bounded workflow explicitly restores native runtime names and refreshes `supporting_assets/DCOIR_Collector.zip`.

Hard rules
- Do not ship a wrapper-only collector package.
- Do not omit the collector part set from the compile step.
- Do not emit raw `.ps1` names in the delivery package when the current mail-filter rule requires `.txt` suffixes.
- Do not overwrite the retained runtime zip directly from the delivery package without a rename/restore step.
- Keep all workflow smoke tests on the same build algorithm used by the manual collector delivery workflow.

Recommended next move
- Use the manifest-driven compile-and-package workflow as the current collector delivery lane, then add a later bounded restore-and-refresh step if you want the workflow to also regenerate `supporting_assets/DCOIR_Collector.zip` from the approved delivery package.
