DCOIR Gemini Legacy Surface Inventory and Hygiene Plan v1.0.0

Purpose
- Inventory the currently known Gemini-related legacy, generated, comparative, and reference surfaces in the repo.
- Clarify what each surface is for so future work does not mistake reference material for the current operator-facing build authority.
- Establish a bounded hygiene plan before any broader move, rename, or archival consolidation is attempted.

Inventory baseline covered by this document
- knowledge/comparative_reference_agent_markdowns/
- knowledge/generated_agent_markdowns/
- project_sources/DCOIR_Gemini_Email_Build_Bundle_v1_1_0/
- project_sources/legacy_reference_gemini_agents/
- project_sources/original_alert_triage_gemini_agents/
- project_sources/original_collector_artifact_gemini_agents/

Role map

1. knowledge/comparative_reference_agent_markdowns/
Role
- Readable comparative-reference working surface.
Use
- Compare current generated output against known-good or historically useful reference sets.
Do not treat as
- control-plane authority
- current operator-facing build authority by itself

2. knowledge/generated_agent_markdowns/
Role
- Readable generated-agent working surface.
Use
- Inspect the latest generated markdown outputs, support packaging, support parity review, and retain a readable generated-output surface in GitHub.
Do not treat as
- design-time behavioral authority
- a replacement for PP-09, PP-10, or the control plane
- the default shipment surface when a fresher generation round is required

3. project_sources/DCOIR_Gemini_Email_Build_Bundle_v1_1_0/
Role
- Extracted legacy bundle retained for structure and naming reference only.
Use
- Study folder shape, naming lessons, and legacy bundle composition choices.
Do not treat as
- current wording authority
- current operator-facing build authority
- current DCOIR Gemini generation source

4. project_sources/original_alert_triage_gemini_agents/
Role
- Historical benchmark set for verbosity, field-writing quality, routing richness, and instruction explicitness.
Use
- Benchmark future Gemini Description and Instructions density and quality.
Do not treat as
- the current DCOIR operator-facing build
- current DCOIR behavioral authority by itself

5. project_sources/original_collector_artifact_gemini_agents/
Role
- Historical failure-reference surface.
Use
- Recognize thin, under-explicit, or otherwise insufficient collector-oriented generated output.
Do not treat as
- a current reference to imitate directly
- current DCOIR operator-facing build authority

6. project_sources/legacy_reference_gemini_agents/
Role
- Governed home for clearly labeled manual or curated legacy Gemini captures with ongoing reference value.
Use
- Preserve verbatim captures or bounded imported legacy materials that are still useful for comparison, salvage, migration, or field-quality benchmarking.
Do not treat as
- current operator-facing build authority unless a later governed change explicitly promotes a file for another purpose

Current authority map to keep separate from legacy/reference surfaces
- Control plane: CP-01 and CP-02
- Current standalone and Gemini governance: DOC-04, DOC-07, PP-09, PP-10, and the current prompt-pack authority line where behavior authority is involved
- Supporting readable working surfaces: generated markdowns, comparative markdowns, knowledge docs, and retained supporting zips

Observed hygiene risks
1. A future worker could mistake generated markdowns for design-time authority.
2. A future worker could mistake extracted legacy bundles for the current operator-facing build.
3. A future worker could collapse the alert-triage benchmark, the collector failure reference, and manual legacy captures into one undifferentiated "old Gemini" bucket.
4. The repo may still contain additional legacy-adjacent files or folders outside the bounded baseline covered here.

Hygiene plan

Phase 1: labeling and role clarity
- Keep additive governed docs that explain the role of each known surface.
- Prefer role clarification before moving files.
- Preserve a clear distinction among current authority, supporting generated surfaces, comparative surfaces, and legacy reference surfaces.

Phase 2: bounded inventory
- Perform a bounded repo pass for other legacy-adjacent Gemini-readable files, folders, or zips in knowledge/, project_sources/, supporting_assets/, and related folders.
- For each additional surface found, assign one explicit role category before any move or rename is attempted.

Phase 3: optional normalization
- Only after the bounded inventory is complete, decide whether broader folder normalization, renaming, or archival grouping would materially reduce confusion.
- Do not perform a broad move or rename based only on cursory discovery.

Allowed role categories for future inventory work
- current control-plane authority
- current generation workflow authority
- readable generated working surface
- readable comparative working surface
- structure-and-naming legacy reference
- quality-and-density benchmark
- failure reference
- manual or curated legacy capture
- retained delivery artifact only

Hard rules
- Do not let a generated surface silently become design-time authority.
- Do not let a legacy bundle silently become current operator-facing authority.
- Do not move broad legacy material into legacy_reference_gemini_agents without a bounded inventory decision.
- Prefer additive explanation and explicit role labeling before structural cleanup.

Best next move
- Continue with a bounded inventory pass for any remaining Gemini-related legacy or generated surfaces not yet covered by this baseline, then decide whether any README or folder normalization changes are truly warranted.
