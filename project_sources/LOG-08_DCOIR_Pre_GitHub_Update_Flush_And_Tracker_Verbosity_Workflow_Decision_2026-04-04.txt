DCOIR Pre-GitHub-Update Flush/Manicure And Tracker Verbosity Workflow Decision - 2026-04-04

Purpose
- Record the approved 2026-04-04 workflow decisions for pre-GitHub-update state flush/manicure, resume-speed requirements, governance-gap handling, and verbose tracker behavior.
- Use `dcoir-session-tracker` as the reference model for how stateful DCOIR helper skills should surface local working state before governed repo updates.

Approved durable workflow decisions
- Before any governed Project update that depends on state held by a stateful helper skill, run a bounded flush/manicure check rather than treating that state as implicitly understood.
- `dcoir-session-tracker` is the reference implementation for this workflow line.
- A valid flush/manicure check must inspect the current local state surface when it exists, surface promotion candidates, surface what should remain local for now, and end with one best next move.
- Fresh-session resume should be quick, predictable, and repeatable. Startup-speed and resume predictability are explicit acceptance-test concerns for this workflow line.
- Governance-gap handling must include an authoritative current-state or version-checking model, drift detection across active control-plane and continuity surfaces, an attention-worthy callout when drift is found, and an explicit remediation path rather than quiet continuation.
- Tracker entries and tracker readouts must be verbose enough that the operator can understand the carried state without reconstructing prior chat context.
- The minimum verbose tracker-entry standard is: title, full detail, why it matters, next action, carry-forward or promotion note when relevant, and related files/skills/artifacts when known.
- Operator-facing tracker output should default to the verbose form. Terse summaries may still exist internally, but they should not be the primary operator-facing representation for materially important items.

Reference flush/manicure sequence
1. Re-anchor to Project Instructions, then CP-01, then CP-02.
2. Inspect the active local session-state surface when one exists.
3. Inventory buffered open items, grouped by session-only state, promotion candidates, durable preference candidates, and blocked or authority-sensitive items.
4. Surface what is safe to promote now, what should remain local for now, what must be exported if the session ends before promotion, and what continuity surfaces would need refresh.
5. If the current workflow is already performing a safe governed repo write, batch the already-known related continuity updates into one grouped transaction.
6. If no safe repo write is happening, keep the state buffered or export it, but do not imply it became durable automatically.
7. End with one best next move.

Implementation posture
- Patch the tracker reference model first, then validate it before widening the same pattern to other stateful skills.
- Treat the local JSON file as real only when inspection proves it.
- Treat the tracker export as a continuity artifact, not as control-plane authority.
- Keep GitHub Desktop or other operator-managed update lanes as execution-lane choices, not as durable governance rules, unless later approved as such.

Immediate validation targets
1. Capture and retain the startup-speed or new-session resume concern in the local tracker and rendered export without ambiguity.
2. Capture and retain the governance-gap concern in the same way.
3. Inspect the real local JSON file and prove the state exists.
4. Render an export that preserves the richer item detail.
5. Perform a fresh-session resume test and judge whether the active work line is recovered quickly and predictably enough.

Best next move
- Run bounded regression on the tracker reference implementation and then tighten the current-state or version-checking enforcement model using the tracker workflow as the proving ground.
