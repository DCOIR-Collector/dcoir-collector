DCOIR Repository Layout Spec 1.0.0

Purpose
- This file defines the canonical repo and deployment layout for DCOIR_Project deliverables.
- It exists to make offline repo generation, deployment packaging, and repeatable local testing deterministic.
- It is separate from the Project control plane so layout changes do not overload the manifest or change log.

Scope
- This spec defines one authoritative layout blueprint with two output modes:
  - Repo mode
  - Full-refresh update mode
- It defines the canonical directory tree for generated repo bundles.
- It defines the full-refresh update-bundle structure for delete-and-reupload syncs.
- It defines mapping rules from uploaded Project-readable sources to native output filenames.
- It defines how supporting package assets such as DCOIR_Collector.zip are placed in runtime-friendly layouts.
- It defines explicit pattern-based handling for Knowledge docs.
- It records the class-prefix naming model for the uploaded Project-readable source set.

Output modes
- Repo mode
  - Use for git-like, offline, local-testing, or deployment-style repo bundles.
  - Generate the canonical hierarchical DCOIR_Project/ tree.
  - Emit native runtime filenames where this spec explicitly requires native emission.
- Full-refresh update mode
  - Use for broad Project refreshes, structural changes, renames, or multi-file syncs where the operator will replace the uploaded source set.
  - Generate a flat uploaded-source refresh bundle using project_sources/, supporting_knowledge_docs/, supporting_assets/, release_notes/, and optional project_settings/.
  - Keep uploaded Project-readable source filenames unchanged in project_sources/ and keep supporting Knowledge docs unchanged in supporting_knowledge_docs/.
  - Treat files omitted from the relevant full-refresh folders as intentionally excluded from the refreshed uploaded workspace set.

Canonical repo-mode layout
- DCOIR_Project/
  - stable/
    - assets/
  - rollback/
  - control_plane/
  - docs/
  - logs/
  - prompts/
  - knowledge/
  - bundles/

Canonical full-refresh update-bundle layout
- update_bundle/
  - project_sources/
  - supporting_knowledge_docs/
  - supporting_assets/
  - release_notes/
  - optional project_settings/
  - optional local_prompts/

Folder intent in repo mode
- stable/
  - current stable operational files needed for local harness testing or deployment packaging
- stable/assets/
  - supporting runtime package assets
  - current supporting package asset: DCOIR_Collector.zip
- rollback/
  - rollback or engine-reference files retained for fallback or comparison
- control_plane/
  - the small governance control plane for the current workspace
- docs/
  - evergreen process and workflow documents
- logs/
  - situational status, handoff, lessons learned, and planning logs
- prompts/
  - modular prompt-pack source files, the promoted combined analyst-facing master prompt runtime source, and related AI workflow source material
- knowledge/
  - supporting human-readable Knowledge docs and companion knowledge files
- bundles/
  - generated update bundles, offline repo ZIPs, and deployment packages

Stored-source naming rules
- Uploaded Project-readable sources remain flat uploaded files in the Project workspace.
- In full-refresh update mode, uploaded Project-readable sources are emitted flat into project_sources/.
- Supporting human-readable Knowledge docs are emitted into supporting_knowledge_docs/ in full-refresh update mode.
- Supporting uploaded package assets are emitted into supporting_assets/ in full-refresh update mode.
- Settings-applied content is emitted into project_settings/ when it must be updated alongside uploaded source changes.
- Uploaded Project-readable source files now use class-prefixed naming for the major authoritative source classes:
  - CP = control plane
  - DOC = evergreen workflow docs
  - LOG = situational logs
  - PP = modular prompt-pack and AI workflow sources
  - ST = stable operational baseline text source
  - RB = rollback reference text source
- When a source file is meant to stay Project-readable but also map cleanly to a native runtime type, use a double extension.
- Current double-extension examples:
  - run_DCOIR_Tests.ps1.txt
  - run_DCOIR_Tests.cmd.txt
- Plain-text control files, logs, prompts, and docs remain single-extension .txt files.
- Native package assets such as .zip remain native and are not converted.
- Knowledge docs use the filename pattern:
  - Knowledge - ## - Title.md.txt
- All files matching Knowledge - ## - *.md.txt are automatically treated as Knowledge docs with no allowlist.

Native output emission rules
- If a filename ends with .ps1.txt, emit the native runtime file by stripping only the terminal .txt.
- If a filename ends with .cmd.txt, emit the native runtime file by stripping only the terminal .txt.
- If a filename ends with .json.txt or .xml.txt in a future sync, emit the native runtime file by stripping only the terminal .txt.
- If a file does not use a double extension, keep the filename unchanged unless this spec explicitly maps it to a native runtime alias.
- Binary or packaged files such as .zip and .pdf remain unchanged.
- Knowledge docs remain unchanged native .md.txt files in both output modes.

Canonical source handling for full-refresh update mode
- project_sources/
  - include the complete uploaded Project-readable source set for the refreshed workspace
  - keep uploaded Project-readable filenames unchanged
- supporting_knowledge_docs/
  - include the complete uploaded supporting human-readable Knowledge-doc set for the refreshed workspace
  - keep Knowledge - ## - *.md.txt filenames unchanged
- supporting_assets/
  - include supporting uploaded package assets such as DCOIR_Collector.zip
  - do not place supporting uploaded package assets in project_sources/ unless a future workflow explicitly changes that rule
- project_settings/
  - include settings-applied content such as SET-01_AFRICOM_SOC_IR_Project_Instructions_v9.txt when the settings baseline must be refreshed alongside uploaded source changes
- release_notes/
  - include RELEASE_INSTRUCTIONS.txt when the bundle contains more than one file
- optional local_prompts/
  - use only when local-only operator prompts are intentionally included in the bundle
