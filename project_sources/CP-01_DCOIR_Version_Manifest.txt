GOVERNANCE MODEL
- Class-prefixed filenames remain the authoritative uploaded Project-readable source model.
- The default control-plane files remain Project Instructions, CP-01_DCOIR_Version_Manifest.txt, and CP-02_DCOIR_Change_Log.txt.
- DOC-01_AFRICOM_SOC_IR_Project_Setup_and_Workflow.txt and DOC-02_DCOIR_Project_Transition_Checklist.txt remain evergreen documents and should change only when their evergreen process materially changes.
- DOC-03_DCOIR_Repository_Layout_Spec_v1_0_0.txt remains the canonical repo and deployment layout blueprint and should change only when the layout or emit rules materially change.
- Exact multi-file release actions belong in release_notes/RELEASE_INSTRUCTIONS.txt inside the release ZIP.
- LOG-01_DCOIR_Todo_Log.txt, LOG-02_DCOIR_Lessons_Learned_Log.txt, and LOG-03_DCOIR_Session_Handoff_Brief.txt should be refreshed only when their own substance changes.
- Supporting Knowledge docs are maintained as one uploaded package asset named supporting_knowledge_docs.zip to reduce Project delete/upload count.
- Workflows, execution steps, and helper skills that need Knowledge content must unzip or otherwise reference supporting_knowledge_docs.zip as needed rather than assuming individual Knowledge docs are uploaded into the Project workspace.
- Comparative reference agent markdowns, preserved original Gemini baseline bundles, generated agent markdowns, and Gemini email/build bundles are supporting assets, not control-plane authority.
- Substantive test sessions that materially change active direction, packaging, design strategy, or confidence discipline must be distilled into governed logs and, when they change the live work line, promoted through a control-plane refresh rather than left only in side artifacts or chat history.
- Hard-limit project-instructions rule: any Project Instructions update must fit within 8000 characters before it is promoted into the settings baseline.
- Project-side dcoir-* helper skills are authoring, QA, packaging, workflow, and maintenance aids for this Project; they are not runtime capabilities of the final combined prompt or Gemini outputs.
- Final standalone combined-prompt and Gemini artifacts must be runtime-self-sufficient and must not mention or assume dcoir-* helper skills, hidden project memory, hidden helper workflows, or unattached workspace files.
- Required attention-signaler use points are first substantive DCOIR response, major milestone, action-required / approval-needed, blocked state, and major completion.
- Multi-file deliverables should default to one ZIP bundle unless release notes explicitly declare a narrower targeted update.
