AFRICOM_SOC_IR Project Setup and Workflow

Purpose
- This document is the evergreen setup and workflow guide for the AFRICOM_SOC_IR / DCOIR project.

Project model
- GitHub repository malwaredevil/dcoir-collector is the sole working source for readable governed text files.
- Project space is the bootstrap and applied runtime anchor, not a second editable readable text repository.
- Project Instructions are applied in settings and remain the first resume anchor.
- When readable governed files are needed after that, use the GitHub connector against malwaredevil/dcoir-collector.

Readable-source rule
- Readable governed collector text now uses a split-source model.
- The canonical emitted runtime collector filename remains project_sources/DCOIR_Collector.ps1.
- The readable governed collector source set is project_sources/collector_parts/*.ps1.
- The runtime collector is a thin loader wrapper that dot-sources the numbered collector parts.
- The harness readable source remains project_sources/run_DCOIR_Tests.ps1.
- The current todo structure now uses a split model anchored by project_sources/LOG-01_DCOIR_Todo_Log.txt, project_sources/LOG-01_DCOIR_Todo_Index.txt, and project_sources/todo/*.txt.
- Extracted readable source folders with ongoing project value should live in governed folders under project_sources/ or knowledge/, not remain trapped inside zip bundles.

Structural refactor standard
- Governed structural refactors for growth-prone readable files follow project_sources/DOC-04_DCOIR_Structural_Refactor_Standard.txt.
- Prefer index-plus-child-file or wrapper-plus-role-bounded-part patterns when a governed file becomes brittle.
- Preserve stable runtime or resume entry files, define split boundaries by role, create dependencies first, cut over stable entrypoints last, and refresh control-plane and continuity references immediately after structural change.

Packaging convention
- For GitHub-maintained readable text updates, prefer direct GitHub edits over patch ZIPs when the connector path supports them.
- Use the GitHub connector directly as the preferred execution surface when DCOIR work needs to read, create, or update governed readable text in malwaredevil/dcoir-collector.
- Before GitHub-family execution work that is likely to have a validated procedure, consult canonical task memory in knowledge/task_memory first. Use a dedicated preflight helper such as dcoir-memory-preflight when it is installed.
- Use dcoir-* helper skills for planning, analysis, impact review, validation support, and workflow support rather than as the default GitHub write lane.
- For existing-file updates, prefer the validated low-level git-object lane through create_blob, create_tree, create_commit, update_ref, and connector readback.
- For multiple related existing-file changes or removals, prefer one bounded multi-file git-object transaction over one-file-at-a-time updates.
- When the current head sha is needed and fetch_commit is flaky upstream, compare_commits with base=main and head=main may be used to recover the current head sha before create_tree and create_commit.
- After grouped GitHub changes, verify every touched path, including absent-path verification for deletions.
- For operator-side bulk file placement, extracted-file ingestion, or binary and zip handling, prefer GitHub Desktop on a local clone as the easiest approved path.
- Use GitHub Web UI only for small or simple operator-managed manual edits. GitHub CLI and GitKraken are optional, not the default operator path.

Helper-skill workflow rules
- Use dcoir-memory-preflight both before high-friction execution and after a blocker or failed attempt is successfully overcome when the lesson could improve a repeatable workflow, reusable limitation note, reusable failure signature, or helper-skill/process guidance.
- Do not silently promote every recovered blocker into canonical memory. First classify whether the lesson is one-off or reusable, then stage a bounded promotion candidate.
- Relevant stateful helper skills may accumulate session-local buffer content during the chat and flush it into GitHub in grouped updates at the next suitable write point rather than writing every small change immediately.
- Session-local buffered content is not cross-session durable unless it is flushed to GitHub or exported into a handoff artifact.
- Preferred flush-check trigger points are before GitHub writes, after blocker resolution, when switching major tasks, at major milestones, before session export or handoff, when the operator asks what remains, and when a helper skill reports meaningful state drift.
- Prefer staging content in-session and using as few GitHub operations as possible.
- If the grouped existing-file update lane cannot yet be completed safely in-chat, keep the grouped-write intent explicit rather than pretending the limitation is solved.
- After every helper-skill create or update, pass the result through dcoir-skill-regression-auditor before treating the skill as ready for broader use or workflow promotion.
- When DCOIR helper-skill source is governed in GitHub under the one-source-of-truth model, the default new-skill rollout workflow is build, download, manual install, extract, and GitHub Desktop source ingest.
- When a DCOIR helper skill already has governed GitHub source, the default update workflow is build, download, manual install, and assistant-managed GitHub source update when the connector lane can do so safely.
