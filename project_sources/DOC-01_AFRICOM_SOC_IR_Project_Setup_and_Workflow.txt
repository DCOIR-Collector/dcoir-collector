AFRICOM_SOC_IR Project Setup and Workflow

Purpose
- This document is the evergreen setup and workflow guide for the AFRICOM_SOC_IR / DCOIR project.

Project model
- GitHub repository malwaredevil/dcoir-collector is the sole working source for readable governed text files.
- Project space is the bootstrap and applied runtime anchor, not a second editable readable text repository.
- Project Instructions are applied in settings and remain the first resume anchor.
- When readable governed files are needed after that, use the GitHub connector against malwaredevil/dcoir-collector.

Readable-source rule
- Readable governed collector text now uses a split-source model.
- The canonical emitted runtime collector filename remains project_sources/DCOIR_Collector.ps1.
- The readable governed collector source set is project_sources/collector_parts/*.ps1.
- The runtime collector is a thin loader wrapper that dot-sources the numbered collector parts.
- The harness readable source remains project_sources/run_DCOIR_Tests.ps1.
- The current todo structure now uses a split model anchored by project_sources/LOG-01_DCOIR_Todo_Log.txt, project_sources/LOG-01_DCOIR_Todo_Index.txt, and project_sources/todo/*.txt.

Packaging convention
- For GitHub-maintained readable text updates, prefer direct GitHub edits over patch ZIPs when the connector path supports them.
- Use the GitHub connector directly as the preferred execution surface when DCOIR work needs to read, create, or update governed readable text in malwaredevil/dcoir-collector.
- Before GitHub-family execution work that is likely to have a validated procedure, consult canonical task memory in knowledge/task_memory first. Use a dedicated preflight helper such as dcoir-memory-preflight when it is installed.
- Use dcoir-* helper skills for planning, analysis, impact review, validation support, and workflow support rather than as the default GitHub write lane.
- For existing-file updates, prefer the validated low-level git-object lane through create_blob, create_tree, create_commit, update_ref, and connector readback.
- For multiple related existing-file changes or removals, prefer one bounded multi-file git-object transaction over one-file-at-a-time updates.
- When the current head sha is needed and fetch_commit is flaky upstream, compare_commits with base=main and head=main may be used to recover the current head sha before create_tree and create_commit.
- After grouped GitHub changes, verify every touched path, including absent-path verification for deletions.
