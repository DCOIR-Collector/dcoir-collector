AFRICOM_SOC_IR Project Setup and Workflow

Purpose
- This document is the evergreen setup and workflow guide for the AFRICOM_SOC_IR / DCOIR project.
- It describes how the project is structured, how resume works under the GitHub-primary model, and how bootstrap updates should be applied.
- It should not be rewritten for every normal release.

Project model
- The project is the current operational workspace, not the full historical archive.
- GitHub repository malwaredevil/dcoir-collector is the sole working source for readable governed text files.
- Project space is the bootstrap and applied runtime anchor, not a second editable readable text repository.
- Project Instructions are applied in settings and remain the first resume anchor.
- When readable governed files are needed after that, use the GitHub connector against malwaredevil/dcoir-collector.
- Do not keep duplicate editable readable text files in both GitHub and Project space.
- Supporting assets can still exist in GitHub and optionally in Project space when they are intentionally retained there.
- Supporting assets do not override the control plane.
- Project-side dcoir-* helper skills are authoring, QA, packaging, workflow, and maintenance aids for this project.
- Final standalone combined prompt and final Gemini outputs must be runtime-self-sufficient and must not mention or assume dcoir-* helper skills, hidden project memory, hidden helper workflows, or unattached workspace files.

Current control-plane model
- The default resume anchor is:
  - Project Instructions
  - GitHub CP-01_DCOIR_Version_Manifest.txt
  - GitHub CP-02_DCOIR_Change_Log.txt
- Use CP-01 to identify what is current.
- Use CP-02 to understand what changed and why.

Readable-source rule
- Readable governed text files live in GitHub, not duplicated in Project space.
- Native readable script formats are preferred in GitHub when the connector can read them.
- Current readable script-source working files are:
  - project_sources/DCOIR_Collector.ps1
  - project_sources/run_DCOIR_Tests.ps1
  - project_sources/run_DCOIR_Tests.cmd
- Prompt-pack, docs, logs, control-plane files, rollback references, and settings mirrors also live in GitHub as the readable working set.
- Knowledge markdown in knowledge/ is the readable working source when present.
- supporting_knowledge_docs.zip may be retained as a delivery or reference asset, but not as the preferred editable readable source when extracted knowledge markdown exists.

How to start or resume work
- Apply the current Project Instructions text from project_settings/SET-06_AFRICOM_SOC_IR_Project_Instructions_v15.txt to the Project settings field.
- In a fresh project chat, resume from Project Instructions first.
- Then use the GitHub connector against malwaredevil/dcoir-collector to read:
  - project_sources/CP-01_DCOIR_Version_Manifest.txt
  - project_sources/CP-02_DCOIR_Change_Log.txt
  - other governed readable sources as needed
- Do not assume duplicate readable Project uploads exist for those files.
- If binary or retained supporting assets are needed and intentionally present in Project space, treat them as supporting assets rather than control-plane authority.

How to apply a future bootstrap update
1. Download the update ZIP bundle and extract it locally.
2. Open release_notes/RELEASE_INSTRUCTIONS.txt inside that bundle.
3. Apply the updated Project Instructions text to the Project settings field.
4. Delete obsolete duplicate readable text files from Project space when release instructions say to do so.
5. Retain only the intentionally kept bootstrap or binary assets in Project space.
6. Resume from Project Instructions in a fresh chat and let the project read the readable governed sources from GitHub.

Packaging convention
- For GitHub-maintained readable text updates, prefer direct GitHub edits over patch ZIPs when the connector path supports them.
- Use ZIP bundles for Project bootstrap/application steps, multi-file delivery, or retained binary/supporting assets when needed.
- If a bundle contains more than one file, include release_notes/RELEASE_INSTRUCTIONS.txt.
- Do not use Project refresh bundles to reintroduce duplicate readable source mirrors unless an explicit temporary exception is declared.

Operational content rules
- Keep Windows PowerShell 5.1 compatibility as a hard requirement unless explicitly changed.
- Use Elastic Defend response-action syntax for endpoint instructions.
- Use PowerShell commands for local workstation and local test instructions.
- Use the runtime or native GitHub filename when discussing execution where applicable.
- When a helper skill changes materially, replace the external skill package separately, verify icon or metadata where expected, rerun post-update invocation or accessibility regression where relevant, and then refresh governed project sources if the durable workflow changed.

Knowledge handling
- Knowledge docs are authored and maintained as normal markdown in GitHub knowledge/.
- supporting_knowledge_docs.zip may remain a delivery or reference asset, but it is not the preferred editable readable source when extracted markdown exists.
- Knowledge docs must not override Project Instructions, CP-01, CP-02, or other authoritative workflow sources.
