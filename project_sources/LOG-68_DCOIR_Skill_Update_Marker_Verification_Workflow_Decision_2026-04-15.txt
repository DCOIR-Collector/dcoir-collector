DCOIR Skill Update Marker Verification Workflow Decision

Date
- 2026-04-15

Decision
- Adopt marker-based installed-skill verification for manual helper-skill update flows.

Why
- Live testing of the helper-skill update path showed that the skill editor could reflect a saved update before assistant-side readback fully caught up across all files.
- That made immediate assistant-only verification too weak to justify GitHub source sync, parity closure, or repo-bundle generation by itself.
- A simple per-file marker gives the operator and assistant one explicit confirmation target in the edited installed file set.

Approved rule
- Remove older `skill-marker:` lines or comments from the edited files before packaging an updated skill.
- Add one fresh `skill-marker:` entry to each edited file using the correct file comment syntax.
- After the operator saves the update, verify the expected current markers in the edited installed file set before treating the installed skill as safe for GitHub source sync, GitHub Desktop repo-update bundle generation, parity closure, or broader readiness claims.
- Treat the skill editor as primary truth when it is available.
- Treat assistant-side readback as secondary and potentially delayed.
- When those two surfaces disagree, keep the state bounded instead of assuming source drift.

- This decision changes helper-skill update verification posture, not the control plane.
