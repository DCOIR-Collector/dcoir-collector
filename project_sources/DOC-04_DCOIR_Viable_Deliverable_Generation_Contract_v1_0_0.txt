DCOIR Viable Deliverable Generation Contract v1.0.0

Purpose
- Capture the durable rules that produced the first operator-acceptable v2 standalone prompt bundle and v2 Gemini email build bundle shape.
- Prevent future drift back into packaging old generated artifacts, under-dense Gemini fields, or standalone prompt rewrites that omit prompt-pack substance.
- Keep the project aligned to explicit validation gates and operator-facing bundle shapes that can be tested directly.

Scope
- This contract governs two deliverable classes:
  1. the standalone analyst-facing all-in-one prompt bundle
  2. the Gemini parent-agent plus seven-sub-agent bundle
- This contract is for deliverable generation and test-facing packaging, not for replacing behavioral authority.

Role hierarchy
1. Behavioral authority
- PP-01 through PP-07 remain the design-time behavioral source of truth.

2. Standalone runtime deliverable
- The shipped standalone prompt must be freshly generated from PP-01 through PP-07 as a true full assembly.
- PP-08 may exist in GitHub as the regenerated runtime parity artifact, but PP-08 is not the design-time source of truth and is not a license to ship a condensed rewrite.

3. Gemini runtime deliverable
- The default operator-facing Gemini bundle now follows a stored-source and compile model.
- `project_sources/DCOIR_Gemini_Bundle_Source/` is the governed editable runtime source tree for the parent agent, sub-agents, and attachment files.
- The shipped Gemini bundle should normally be compiled from that source tree rather than regenerated on the fly.
- PP-09 remains the Gemini workflow source and PP-10 remains the bounded design artifact for designing, refreshing, or structurally changing the Gemini line, but they are not by themselves the runtime shipment surface.

4. Gold-standard style and density benchmark
- `project_sources/original_alert_triage_gemini_agents/` is the approved style, density, routing, and field-writing benchmark.
- Its Description and Instructions posture is the minimum acceptable baseline for future Gemini refreshes.
- Future Gemini source-tree updates should be at least as explicit, routing-rich, and operationally dense as that benchmark set.

5. Structure and naming reference only
- `project_sources/DCOIR_Gemini_Email_Build_Bundle_v1_1_0/` is an approved structure and naming reference only.
- Use its bundle shape, folder model, and naming lessons when they help the operator-facing delivery shape.
- Do not treat its wording or agent text as current quality authority.

6. Failure-reference surfaces
- `project_sources/original_collector_artifact_gemini_agents/` is a historical failure reference for what not to ship when the generated fields are too thin.
- Earlier stored generated markdowns are generated-output working surfaces only. They are not the default shipment surface unless the operator explicitly asks for the stored version.

Standalone prompt-generation rules
- The standalone prompt must be a true full prompt-pack assembly from PP-01 through PP-07.
- Do not ship a condensed rewrite that drops material from the modular source line.
- Attachments may support runtime use, but they must not substitute for omitted prompt-pack substance.

Gemini stored-source rules
- Keep one governed editable source tree for the accepted runtime Gemini bundle.
- The source tree should contain the parent agent, each sub-agent, start-here files, and the approved knowledge attachment set.
- Routine bundle downloads should compile this stored source tree directly.
- If requirements change, edit the relevant source-tree files first, then compile.
- Only use on-the-fly generation when the operator explicitly wants a redesign, topology change, or major rewrite path before those results are promoted into the stored source tree.
- Do not default to packaging `knowledge/generated_agent_markdowns/` as the runtime bundle.
- Do not substitute governance docs, validation docs, or helper-workflow notes for the approved knowledge attachment set.

Packaging and clarity rules
- Keep the shipped test-facing bundles minimal and operator-facing.
- Do not mix control-plane logs, packager reports, helper-workflow notes, collector internals, harness files, or other unrelated repo material into the Gemini or standalone test bundles by default.
- Keep agent-definition files and attachment files in clearly separate folders.
- Add `.txt` to mail-sensitive markdown or zip-like payload names when the operator needs the bundle to pass email filters.

Hard validation gates
Reject a Gemini bundle if any of these are true:
- the runtime bundle was not compiled from `project_sources/DCOIR_Gemini_Bundle_Source/` during ordinary shipment
- the parent or any sub-agent is materially thinner or less explicit than the approved alert-triage benchmark set
- Description is not routing-rich enough to support delegation well
- Instructions are not operationally explicit enough to preserve workflow, error handling, stop conditions, and output contract
- extra confusing materials are mixed into the main build folder
- the output depends on hidden project helpers or hidden memory
- the approved knowledge attachment set was substituted with governance or validation files

Reject a standalone prompt if any of these are true:
- the all-in-one prompt is a condensed rewrite rather than a true full prompt-pack assembly
- attachments are being used to replace missing prompt-pack substance
- the prompt omits alert triage, escalation logic, IOC enrichment, artifact review, or final synthesis behavior that exists in the modular source line

End state
- The durable success condition is repeatable generation of:
  1. a fresh standalone all-in-one prompt that truly preserves prompt-pack substance
  2. a compiled Gemini parent-plus-seven-sub-agent bundle produced from the governed stored source tree
- Both deliverables must be shippable for operator testing without requiring reconstruction from buried chat history.
