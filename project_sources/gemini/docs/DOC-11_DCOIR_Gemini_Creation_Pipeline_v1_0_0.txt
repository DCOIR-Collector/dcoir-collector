DCOIR Gemini Creation Pipeline v1.1.0

Purpose
- Define the governed create-and-ship path for the operator-facing Gemini bundle after the stored-source compile model was adopted.
- Make the Gemini bundle manifest the explicit source of topology truth for the prime agent and sub-agent set.
- Keep future topology changes governed, reviewable, and repeatable.

Current authority and source model
- PP-01 through PP-07 remain design-time behavioral authority.
- `project_sources/gemini/bundle_source/` remains the editable runtime source tree.
- `Gemini_Bundle_Source_Manifest.json` remains the compile contract and now also carries the explicit topology contract.
- Ordinary shipment remains validation plus compilation, not ad hoc regeneration.

Topology truth rules
- The manifest is the explicit source of truth for:
  - the prime-agent file
  - the listed sub-agent files
  - the expected sub-agent count
  - the required-file set that must ship
- Validation must compare the manifest topology against the actual source-tree contents and fail on drift.
- Workflow required-surface validation must read Gemini bundle required files from the manifest rather than duplicating hardcoded sub-agent file lists.

Required behavior when adding or removing sub-agents
1. Edit the relevant runtime source files in `project_sources/gemini/bundle_source/`.
2. Update the manifest topology section.
3. Update `required_files` in the manifest.
4. Update the generated index and quick-start guidance when the build instructions or topology changed.
5. Re-run validation and build.
6. Do not treat a new source file as active topology until the manifest and index both reflect it.

Workflow
1. Confirm the current control plane.
2. Edit the affected source files in `project_sources/gemini/bundle_source/`.
3. Update the manifest topology and required-file contract when the topology changed.
4. Run `validate_dcoir_gemini_bundle.py`.
5. Run `build_dcoir_gemini_release.py`.
6. Review the workflow summary and uploaded build artifacts.
7. Ship the compiled zip only after the validation path passes.

Hard rules
- Do not let folder discovery silently replace the manifest as topology authority.
- Do not add or remove sub-agents by file drop alone.
- Do not let the workflow report a successful bundle when manifest topology and source-tree contents drift.
- Keep the validation summary explicit enough that the operator can see the manifest-listed and discovered sub-agent counts.

Recommended next move
- Use the manifest-driven topology contract for the larger major-version Gemini runtime bundle, then validate the current manifest-listed sub-agent topology through the associated Actions workflow before tomorrow's functionality test.
