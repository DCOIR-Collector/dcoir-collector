# Agent Attachment Map

This map is intentionally explicit. It exists so the operator, the maintainer, and the future validation workflow can see exactly which shared knowledge files feed the prime-agent attachment set and why they matter.

1. Knowledge - Core - Overview and About
- Purpose: define current DCOIR posture, source classes, governance boundaries, and the working relationship between the collector line, the Gemini line, and the maintained knowledge set.

2. Knowledge - Core - Elastic Quick Start
- Purpose: preserve endpoint response-console usage, local-versus-endpoint command-lane separation, quick-start action patterns, and endpoint-side guardrails.

3. Knowledge - Collector - Local Test and Regression
- Purpose: preserve local harness philosophy, suite selection, restaging behavior, manual validation standards, and practical regression use from the current repo-style layout.

4. Knowledge - Core - Tier 1 Collect Runbook
- Purpose: preserve first-pass baseline collection posture, outputs, review order, and escalation logic from baseline into narrower next steps.

5. Knowledge - Core - Tier 2 Collect Runbook
- Purpose: preserve deeper-collection logic, persistence-and-configuration context, and decision rules for when deeper context is justified.

6. Knowledge - Core - Enrichment Actions
- Purpose: preserve the one-action-at-a-time enrich model, action-family selection, finalize behavior, retrieval expectations, and tool-backed action discipline.

7. Knowledge - Core - Artifact Review Guide
- Purpose: preserve evidence-first artifact review order, wrapper-versus-evidence-carrier distinctions, upload priority logic, and retrieved-artifact interpretation discipline.

8. Knowledge - Core - Troubleshooting
- Purpose: preserve bounded troubleshooting rules, lane-specific failure diagnosis, packaging and runtime triage habits, and non-overreach behavior.

9. Knowledge - Core - FAQ
- Purpose: preserve repeated operator answers about workflow, packaging, authority, command lanes, and runtime-maintenance habits in a fast-reference format.

10. Knowledge - Gemini - AI Prompt and Agent Design
- Purpose: preserve the current prompt-pack and Gemini design posture, including description-field routing quality, instruction density, comparative-reference usage, and stored-source compile rules.

11. Knowledge - Core - IOC Enrichment and Public Sources
- Purpose: preserve evidence-grounded IOC enrichment policy, source-tier distinctions, public-corroboration boundaries, and per-indicator-type guidance.

12. Knowledge - Gemini - Runtime Bundle and Source Tree
- Purpose: preserve stored-source compile behavior, source-tree layout, manifest-driven inventory thinking, and the practical rules for keeping the runtime bundle coherent.

13. Knowledge - Gemini - Agent Topology and Routing
- Purpose: preserve the current one-prime-plus-eleven-sub-agent runtime topology, the ownership of each sub-agent, and the routing rules that keep responsibilities explicit.

14. Knowledge - Gemini - Output Contract and Command-Lane Discipline
- Purpose: preserve final response-structure rules, decision-state discipline, singular-command behavior, and endpoint-versus-local command-lane separation.

15. Knowledge - Collector - EXE Usage and Runtime Behavior
- Purpose: preserve optional EXE usage, GitHub Actions build-and-validation flow, EXE-versus-PS1 runtime differences, and EXE-aware FailureGates behavior so Gemini and operators do not misread wrapper-limited behavior as a collector regression.

16. Knowledge - Collector - Feature and Output Contract Reference
- Purpose: preserve the canonical collector feature map and high-level output contract so feature descriptions, validation expectations, and Gemini interpretation do not drift across runbooks, troubleshooting, FAQ, and EXE guidance.

17. Knowledge - Reference - Elastic Field Name Reference
- Purpose: preserve the governed exact Elastic field-name reference so Gemini can use a maintained attachment for exact KQL and ESQL field-name construction instead of relying on hard-coded prompt lists or memory.

18. Knowledge - Reference - Elastic Response Actions Reference
- Purpose: preserve the governed native Elastic response-action command reference so Gemini can distinguish native response actions from execute-wrapped shell commands without relying on memory.

19. Knowledge - Reference - OSQuery Reference Index
- Purpose: preserve the routing map that tells Gemini which sharded OSQuery reference page to consult first by investigative objective.

20. Knowledge - Reference - OSQuery Process and Execution Tables
- Purpose: preserve the exact OSQuery process, execution, handle, and runtime-artifact table reference.

21. Knowledge - Reference - OSQuery File and Filesystem Tables
- Purpose: preserve the exact OSQuery file, hash, filesystem, mount, and file-event table reference.

22. Knowledge - Reference - OSQuery Network and Connection Tables
- Purpose: preserve the exact OSQuery network, DNS, interface, route, and socket table reference.

23. Knowledge - Reference - OSQuery User, Auth, and Account Tables
- Purpose: preserve the exact OSQuery user, auth, login, SSH, and account-context table reference.

24. Knowledge - Reference - OSQuery Persistence and Startup Tables
- Purpose: preserve the exact OSQuery persistence, startup, service, scheduled-task, and shim table reference.

25. Knowledge - Reference - OSQuery System, Hardware, and Platform Tables
- Purpose: preserve the exact OSQuery system, hardware, memory, uptime, USB, and platform table reference.

26. Knowledge - Reference - OSQuery Security, Detection, and Event Tables
- Purpose: preserve the exact OSQuery security, firewall, audit, YARA, and event table reference.

27. Knowledge - Reference - OSQuery Application, Package, and Extension Tables

## Maintenance rule

The maintained source is `knowledge/*.md`. The build process packages those files directly into `02_PRIME_AGENT_ATTACHMENTS/*.md.txt` inside the generated release ZIP by appending `.txt` to the source filename. Do not maintain duplicate Knowledge attachment source files under `bundle_source`. `DCOIR_KNOWLEDGE_INDEX.md` is a repo-level operator entry point and is not part of the Gemini attachment set unless the manifest is intentionally expanded.
