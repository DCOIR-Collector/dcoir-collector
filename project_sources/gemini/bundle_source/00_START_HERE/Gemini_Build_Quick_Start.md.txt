# Gemini Build Quick Start

This quick-start file is intentionally verbose because the operator-facing Gemini bundle is expected to remain predictable under mixed evidence and mixed workflow starting points. Use this file as the first operator guide for building, reviewing, and testing the major-version Gemini bundle.

## What changed in this major-version wave

This major-version build expands the bundle from a narrower alert-triage-plus-collector-orchestration posture into a more explicit end-to-end investigation assistant. The active source tree now assumes:
- the operator may start from an alert
- the operator may start from collector output
- the operator may start from retrieved artifacts
- the operator may start from an IOC package in mixed format
- the operator may need a targeted collection design instead of a broad baseline
- the operator may need a much richer shared attachment set that explains both the collector line and the Gemini runtime line in stand-alone detail

The active agent topology now includes ten sub-agents:
1. Session readiness and intake
2. Environment, coverage, and schema mapping
3. Alert-family classification and known-benign-technology differentiation
4. Evidence, provenance, and timeline analysis
5. Query planning and syntax guard
6. DCOIR collector execution and bundle workflow orchestration
7. DCOIR collector artifact interpretation and report extraction
8. IOC parsing and evidence-grounded public enrichment planning
9. Targeted collection design and evidence-gap reduction
10. Output-contract consistency guard and report composition

The active shared attachment set now includes fifteen knowledge pages:
- 01 through 11 preserve the collector, workflow, troubleshooting, FAQ, and prior Gemini posture material
- 12 through 15 preserve Gemini runtime source-tree, topology, output-contract, validation, and attachment-maintenance guidance

## Starter prompts

Starter Prompt 1:
"I am pasting an Elastic alert and I want the single best next triage command. Assume I can run KQL, ESQL, and Elastic response actions. Follow the exact continue format and do not skip the singular command."

Starter Prompt 2:
"I already ran the collector and I am uploading the DCOIR upload summary and the analyst follow-up queue. Interpret what the collector is telling us, tell me the best next file or artifact to upload, and keep the case in collector-aware mode."

Starter Prompt 3:
"I have a mixed IOC package in CSV, PDF text, and copied notes. Parse the indicators, classify them by type, show them in analyst-facing grouped form, then give me the narrowest useful KQL, ESQL, response-action, and collector follow-up options."

## Build process

1. Edit the maintained `knowledge/*.md` files first when the shared knowledge layer changes.
2. Edit the agent source files in `01_GEMINI_AGENT_BUILD/` when behavior changes.
3. Update the manifest required-files inventory whenever the attachment set changes.
4. Update `00_START_HERE/Agent_Attachment_Map.md.txt` whenever the attachment inventory or the purpose of an attachment changes.
5. Rebuild through the governed GitHub Action or the governed build script.
6. Review the build artifact and its JSON reports before treating the bundle as test-ready.

## Validation focus

Every serious review of the current bundle should explicitly cover:
- starter prompt behavior
- collector artifact interpretation
- mixed-format IOC parsing
- targeted collection recommendations
- false-positive-aware handling of known enterprise products
- SINGULAR TRIAGE COMMAND consistency
- leadership-writeup behavior
- attachment-map, manifest, and maintained-knowledge alignment
- ordinary direct generation/build behavior when the shared knowledge set changes
