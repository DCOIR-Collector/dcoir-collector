### Data schema and field reference

```text
Use field references as guides, not guarantees. Prefer actual observed fields from the alert, query result, or uploaded artifact.

Elastic field reference source:
1. Use exact provided field names first.
2. When exact known Elastic field names are needed and the evidence does not already supply them, use Knowledge - 99 - Elastic Field Name Reference.
3. Treat Knowledge - 99 as a governed exact-name reference, not proof that every field exists in every dataset or time range.

Field-use rules:
1. If a field has both analyzed and keyword variants, choose the variant suitable for exact matching.
2. Avoid over-specific field names when the source evidence only supports a broader one.
3. If a query depends on a field not shown in evidence, state that dependency.
4. Do not hard-code a schema from memory when the current result shows a different schema.
5. If the returned data conflicts with a remembered or attached field name, adapt to the returned data and state the difference.

CSV/XLSX and report fields:
1. Preserve original column names when parsing uploaded reports.
2. Do not rename or normalize columns silently when the exact wording matters.
3. If columns are missing, state the missing columns and how that affects confidence.
4. For USB reports, preserve fields needed for NIPR/SIPR separation, reporting window, SNOW-prefix classification, user/device identification, and correction tables.

Collector artifact fields:
1. Use collector report labels and sections as written.
2. Do not merge separate collector files into one source unless they were generated by the same run and scope is clear.
3. Preserve host, timestamp, collection mode, artifact path, and section names when interpreting output.
```
