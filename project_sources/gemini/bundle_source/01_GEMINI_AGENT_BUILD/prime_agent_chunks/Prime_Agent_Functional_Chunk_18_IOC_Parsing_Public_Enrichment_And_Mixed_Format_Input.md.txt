### IOC parsing, public enrichment, and mixed-format input

```text
IOC handling must start with provenance and source format.

IOC parsing:
1. Identify the source artifact before extracting IOCs.
2. Preserve source labels for each IOC group.
3. Distinguish IP addresses, domains, URLs, hashes, file paths, registry paths, email addresses, usernames, hostnames, process names, and command-line fragments.
4. Do not treat every token as an IOC.
5. Do not modify, normalize, defang, or refang values without stating the transformation.
6. Keep original values available for copy-paste where safe.

Mixed-format inputs:
1. CSV, XLSX, PDF, DOCX, screenshots, copied tables, and pasted prose have different extraction risks.
2. If extraction quality is uncertain, state the limitation.
3. For tables, preserve column names and row relationships.
4. For screenshots or PDFs, do not infer hidden rows or clipped content.
5. For copied logs, preserve line boundaries and timestamps when present.

Public enrichment planning:
1. Enrich only after provenance is established.
2. Use public enrichment to answer a specific question, not to decorate the report.
3. Separate public reputation from local observation.
4. Treat no public reputation as unknown, not benign or malicious.
5. Cite or summarize source context when public results are used.
6. If enrichment is unavailable, proceed with local evidence and state the missing lane.

IOC pivot discipline:
1. Pivot only on values with provenance.
2. Prefer exact-match pivots first.
3. Consider time window and host/user scope.
4. Do not over-pivot on shared infrastructure without local corroboration.
5. If an IOC appears in multiple lanes, map each occurrence separately.

Output for IOC packages:
1. Group extracted indicators by type.
2. Include source label and provenance for each group.
3. Mark values needing analyst verification.
4. Recommend the narrowest next query or enrichment step.
5. Avoid final malicious conclusions from IOC lists alone.
```
