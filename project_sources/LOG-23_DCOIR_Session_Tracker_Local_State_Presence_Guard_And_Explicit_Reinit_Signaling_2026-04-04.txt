DCOIR Session Tracker Local-State Presence Guard And Explicit Reinit Signaling - 2026-04-04

Current state id: 2026-04-04-session-tracker-presence-guard-and-delivery-preference-v1

Purpose
- This log records the second dedicated follow-on pass on `dcoir-session-tracker`, focused on the trust gap where the local session-state file could be absent during the same working line without strong enough operator-visible signaling.

Decision summary
- Treat silent local-state absence or silent local-file re-creation as a trust defect, not as a harmless implementation detail.
- Keep `inspect` as the strict proof surface for a pre-existing local file instead of weakening it into a silent recovery path.
- Add a first-use presence preflight and explicit operator-visible re-init signaling on local write paths that initialize a new file because no pre-existing file was present.

What changed in this pass
- The local state store now supports an explicit `ensure-state` first-use preflight.
- The local state store now reports whether the local JSON file was already present or had to be initialized for the current branch.
- Local write paths that can initialize a new file now emit explicit absence or re-init status instead of silently continuing.
- The local-state, session-buffer, and session-closeout workflow guidance now says plainly that a newly initialized file is not proof that the missing interval remained file-backed.

Why it matters
- The operator should not have to infer whether later tracker captures were still backed by the real local JSON file after an unseen absence event.
- A recreated file can preserve work going forward, but it does not retroactively prove that the missing interval was protected.
- Trust depends on separating proof of prior local continuity from honest recovery after local-state absence.

Validation boundary
- This pass is locally validated on a bounded fixture path for:
  - first-use `ensure-state` with no pre-existing file
  - repeated `ensure-state` when the file already exists
  - normal capture when the file already exists
  - capture after forced file absence
  - derived pre-push review after forced file absence
- The next step is a live validation on a real grouped governed push to confirm that the new presence guard and explicit re-init signaling are useful in practice and not merely correct in a lab run.

Next immediate move
- Use the updated tracker on the next real grouped governed push and verify that the presence preflight, explicit re-init signaling, staged governed updates, staged todo actions, and post-push cleanup are strong enough for operator trust.
