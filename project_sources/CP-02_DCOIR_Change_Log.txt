DCOIR Change Log

[CURRENT]
Version: 3.9.71
Current state id: 2026-04-08-gemini-stored-source-compile-model-v1
Change class: Gemini bundle stored-source and compile model adoption

What changed
- Added a new governed runtime Gemini source tree at `project_sources/DCOIR_Gemini_Bundle_Source/` seeded from the accepted good v2 Gemini bundle.
- Added `Gemini_Bundle_Source_Manifest.json.txt` to define the source-tree bundle identity, version, and required files.
- Added `project_sources/generation_validation/compile_dcoir_gemini_bundle.py` so ordinary Gemini downloads can be compiled from the stored source tree.
- Added `DOC-10` to define the stored-source and compile strategy.
- Updated `DOC-04` and `PP-09` so the default Gemini shipment path is now stored-source compilation rather than on-the-fly bundle generation.
- Updated current-state and continuity surfaces to point at the new strategy.

Why
- The accepted good Gemini bundle proved that the runtime-grade agent files and knowledge attachments need to be preserved directly, not approximated from thin generated surfaces at packaging time.
- The project needed a Gemini maintenance path that behaves more like the collector source line and the prompt-pack-to-standalone assembly line.
- The operator explicitly requested a shift from generate-on-the-fly behavior to a save-and-compile model.

Validation notes
- This change installs the source tree and compile path; it does not claim that all future requirements have already been propagated into those source files.
- The next runtime bundle refresh should come from compiling the stored source tree after any desired file edits land there first.
