Current state id: 2026-04-05-package-cleanliness-enforcement-v1

What changed
- Added `project_sources/LOG-31_DCOIR_Package_Cleanliness_Enforcement_And_Clean_Skill_Reissue_2026-04-05.txt` as the governed decision record for the bounded package-cleanliness enforcement pass across the helper-skill library.
- Updated `dcoir-parity-verifier` so packaged skill verification now reports package-cleanliness status and explicit runtime-residue entries such as `__pycache__/`, `*.pyc`, and `.DS_Store` instead of hiding residue completely behind parity exclusions.
- Updated `dcoir-skill-regression-auditor` so regression planning now requires a package-cleanliness check for delivered skill zips in addition to package validation, success-path, failure-gate, and artifact verification.
- Reissued clean install packages for the affected skills found contaminated in the campaign audit: `dcoir-change-impact-analyzer`, `dcoir-collector-qa`, `dcoir-knowledge-doc-maintainer`, `dcoir-source-authority-auditor`, and `dcoir-validation-orchestrator`, together with the updated `dcoir-parity-verifier` and `dcoir-skill-regression-auditor` packages.

Why
- The all-skills campaign found that a subset of delivered skill packages could include runtime residue such as `__pycache__/` and `*.pyc`, which is unnecessary packaging contamination even when normalized parity hashes still match after exclusions.
- The project needed a durable detection and regression rule, not just one-off cleaned reissues.
- The highest-value bounded fix was to harden the parity-verification and regression-auditing skills and then reissue the affected packages cleanly in the same pass.

Validation notes
- A broad package audit across the current helper-skill library confirmed that the contamination was limited to a bounded subset of skill packages rather than the entire library.
- The updated `dcoir-parity-verifier` and `dcoir-skill-regression-auditor` skill packages pass clean packaging validation with no runtime-residue entries included.
- Clean reissued packages were built for the affected skills with no `__pycache__/`, `*.pyc`, or `.DS_Store` entries present in the delivered zips.
- The next immediate branch is to continue the workflow-state high-risk wave from the now live-validated startup, repaired delivery, continuous-execution, and package-cleanliness baseline.
