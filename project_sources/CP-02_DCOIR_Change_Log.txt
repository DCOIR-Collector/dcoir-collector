Change class: governed GitHub helper-skill source rollout and continuity refresh

What changed
- Settled the previously open policy question and approved that all `dcoir-*` skill source should be stored in GitHub.
- Added `project_sources/LOG-06_DCOIR_GitHub_Skill_Source_Policy_Decision_2026-04-03.txt` as the dated project-side decision record for the GitHub helper-skill source policy.
- Added `project_sources/DOC-06_DCOIR_GitHub_Helper_Skill_Source_Layout_And_Rollout.txt` as the evergreen layout and rollout companion for governed helper-skill source.
- Added the governed helper-skill source root `dcoir_skills/` and landed the full governed GitHub source set for all current `dcoir-*` skills.
- Preserved the distinction between governed helper-skill source under `dcoir_skills/` and helper working-state continuity under `dcoir_skill_memory/`.
- Captured the operator preference that when policy, authority, or governance materially blocks progress, the assistant should report the block immediately, explain what would unblock it, and give the best available courses of action.
- Refreshed continuity surfaces so the control plane, handoff view, and dated resume anchor reflect the live `dcoir_skills/` governed source root.

Why
- The project had reached the point where helper-skill source was important enough to govern directly in GitHub instead of leaving the storage model as an open design question.
- The five-skill workflow-enforcement upgrade and the broader helper-skill estate are easier to maintain when GitHub is the readable source-of-truth.
- The operator prefers being offered GitHub Desktop with a repo-ready bundle as a first-class option for large multi-file repo updates.
- Immediate blocked-state escalation improves workflow speed by making operator authorization part of the normal recovery path instead of a late discovery.

- The governed source root `dcoir_skills/` is live in GitHub.
