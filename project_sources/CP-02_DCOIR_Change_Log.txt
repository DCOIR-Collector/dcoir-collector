DCOIR Change Log

[CURRENT]
Version: 3.9.23
Change class: pre-GitHub-update flush/manicure workflow definition and verbose session-tracker upgrade

What changed
- Defined a bounded pre-GitHub-update flush/manicure workflow for stateful helper-skill session state, using `dcoir-session-tracker` as the reference implementation.
- Embedded startup-speed, fresh-session resume predictability, and governance-gap detection into the active workflow line as explicit acceptance-test concerns instead of leaving them as informal chat observations.
- Added a verbose tracker-entry standard so operator-facing tracker items now preserve fuller detail, rationale, carry-forward context, and promotion intent by default.
- Updated the governed `dcoir-session-tracker` skill instructions, local-state workflow reference, export schema guidance, and local scripts so the tracker can store and render richer item detail.
- Refreshed the active todo and continuity surfaces so they now point at post-patch validation of the tracker reference model and the linked current-state or version-checking enforcement problem.

Why
- New-session resume should be quick, predictable, and repeatable rather than dependent on manual reconstruction from terse continuity notes.
- The tracker needs richer stored and rendered state so the operator can understand what is being carried forward without guessing what a short summary means.
- Governance drift remains a critical project risk, so the flush/manicure workflow now explicitly surfaces promotion candidates, remain-local state, and the enforcement path that still needs to be finalized.

Validation notes
- Local tracker regression passed the bounded script path: initialize local state, upsert verbose items, inspect the real local JSON file, and render a markdown export with richer item detail.
- The patched tracker reference model is ready for bounded fresh-session resume and governance-gap regression next.

[PRIOR]
Version: 3.9.22
Change class: local-only session-tracker memory model and continuity refresh

What changed
- Removed the GitHub-backed helper-memory branch from `dcoir-session-tracker` and made the local JSON file the only tracker working-state surface for this skill.
- Refreshed the tracker skill instructions and references so they now route cross-session continuity through exported handoff artifacts or governed Project-file promotion instead of `dcoir_skill_memory/dcoir-session-tracker/session_tracker_state.md`.
- Prepared deletion of `dcoir_skills/dcoir-session-tracker/references/github_memory_workflow.md` and `dcoir_skill_memory/dcoir-session-tracker/session_tracker_state.md` as part of the local-only model transition.
- Refreshed the active todo and continuity surfaces so they no longer describe GitHub-backed tracker memory as an open design branch.

Why
- The tracker now has a real inspectable local JSON state file, so the extra GitHub-backed branch for this skill became unnecessary redundancy.
- Cross-session continuity for this skill should come from exported handoff artifacts or governed Project-file promotion, not from a parallel helper-memory surface.
- The project should stop carrying contradictory language about whether tracker GitHub memory still exists.

Validation notes
- The bounded regression for the local-only model passed the tested path: capture item, inspect local file, ask what remains, trigger move-to-another-session behavior, and verify close-out output against the local file.
- The local file and regression artifacts proved the tracker can operate without a GitHub-backed helper-memory branch.
