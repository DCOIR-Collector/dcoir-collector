DCOIR Change Log

[CURRENT]
Version: 3.9.38
Current state id: 2026-04-04-session-tracker-presence-guard-and-delivery-preference-v1
Change class: session-tracker local-state presence guard, explicit re-init signaling, and delivery-bundle preference capture

What changed
- Added `project_sources/LOG-23_DCOIR_Session_Tracker_Local_State_Presence_Guard_And_Explicit_Reinit_Signaling_2026-04-04.txt` as the governed decision record for the second dedicated `dcoir-session-tracker` pass focused on silent local-state absence risk.
- Added `project_sources/LOG-24_DCOIR_Delivery_Bundle_Structure_And_GitHub_Desktop_Manual_Update_Preference_2026-04-04.txt` as the governed operator-preference record for delivery-bundle structure and GitHub Desktop manual repo-update layout.
- Updated `dcoir-session-tracker` so `scripts/session_state_store.py` now provides an explicit `ensure-state` first-use preflight that reports whether a real local session-state file was already present or had to be initialized for the current branch.
- Updated `dcoir-session-tracker` so its local write paths now emit explicit operator-visible absence or re-init status when a new local session-state file had to be initialized because no pre-existing file was present at that step.
- Kept `inspect` as the strict proof surface instead of silently recreating a missing file during inspection.
- Updated the local-state, session-buffer, and session-closeout workflow guidance so a newly initialized local file is not treated as proof that the missing interval remained file-backed.
- Captured the operator preference that delivery should include only the affected skills or affected repo files unless a broader bundle is explicitly requested.
- Captured the operator preference that single-skill delivery should provide only that one skill zip, while multi-skill delivery should use one outer zip with only the affected top-level per-skill zip files named after the live skills and no subfolders.
- Captured the operator preference that GitHub Desktop manual repo-update bundles should unzip directly to only the affected repo-relative folders and files so they can be dragged into the repo working tree without a wrapper root.
- Refreshed the active continuity and todo surfaces so the next immediate live branch now validates the presence preflight and explicit re-init signaling alongside the stronger pre-push review behavior.

Why
- The operator identified a trust-critical gap: the local session-state file appears to have been absent during the same working line, but the workflow did not surface that strongly enough to show whether later tracker captures were still backed by the real local file.
- The project needed an explicit presence guard and explicit re-init messaging rather than only stronger downstream staging behavior.
- The operator also wanted the preferred delivery structures recorded durably so future packaging work does not need repeated clarification.

Validation notes
- Local bounded regression passed the updated `dcoir-session-tracker` path for first-use `ensure-state`, repeated `ensure-state`, normal capture with an existing file, capture after forced file absence, and derived pre-push review after forced file absence.
- Updated skill package passed local packaging validation.
- Corrected local delivery bundles were rebuilt to the recorded operator-preference structures for both multi-skill delivery and GitHub Desktop manual repo-update use.
- The next immediate branch is a live workflow validation of the updated tracker presence guard and re-init signaling on a real grouped governed push.

[PRIOR]
Version: 3.9.37
Current state id: 2026-04-04-session-tracker-deep-dive-v1
Change class: session-tracker deep-dive pass 1 and derived pre-push review / todo-sync support

What changed
- Added `project_sources/LOG-22_DCOIR_Session_Tracker_Deep_Dive_Pass1_2026-04-04.txt` as the governed decision record for the first dedicated `dcoir-session-tracker` deep-dive pass.
- Updated `dcoir-session-tracker` so it now derives a pre-push review bundle from current local state, including staged governed updates, staged todo actions, and post-push cleanup, instead of relying only on manually staged entries.
- Updated `dcoir-session-tracker` so its operator-facing contract now requires an immediate explicit capture confirmation for materially important items when a real local session-state file exists.
- Updated the tracker export and local-state schema so staged todo actions and the derived pre-push review output are visible in exported or inspected state.
- Refreshed the active continuity and todo surfaces so they now point at live validation of the updated session-tracker pre-push review behavior as the next immediate deep-dive step.

Why
- The operator repeatedly observed that tracker state could be preserved locally without producing a strong enough operator-visible confirmation or a derived staged-governed-update / todo-sync bundle before a real push.
- The workflow docs already expected that behavior, but the implementation was still too manual.

Validation notes
- Local bounded regression passed the updated `dcoir-session-tracker` state-store path for item capture, derived pre-push review generation, staged todo-action creation, export rendering, and post-push staging counts.
- Updated skill package passed local packaging validation.
- The next immediate branch is a live workflow validation of the updated tracker behavior on a real grouped governed push.
