DCOIR Package Cleanliness Enforcement And Clean Skill Reissue - 2026-04-05

Current state id: 2026-04-05-package-cleanliness-enforcement-v1

Purpose
- Record the bounded package-cleanliness enforcement pass across the DCOIR helper-skill library after the coordinated campaign audit found runtime residue in a subset of delivered skill zips.

Decision summary
- Treat runtime residue such as `__pycache__/`, `*.pyc`, and `.DS_Store` inside delivered skill zips as packaging contamination that must be reported explicitly instead of being hidden entirely behind parity exclusions.
- Update `dcoir-parity-verifier` so packaged skill verification now reports package-cleanliness status and explicit runtime-residue entries in addition to the existing normalized tree-hash parity evidence.
- Update `dcoir-skill-regression-auditor` so package-cleanliness checks become a required regression item for delivered skill zips during post-patch validation and coordinated campaign follow-through.
- Reissue clean install packages for the bounded affected set found in the campaign audit: `dcoir-change-impact-analyzer`, `dcoir-collector-qa`, `dcoir-knowledge-doc-maintainer`, `dcoir-source-authority-auditor`, and `dcoir-validation-orchestrator`, together with the updated `dcoir-parity-verifier` and `dcoir-skill-regression-auditor` packages.

Why it matters
- A packaged skill can still look parity-clean after normalized exclusions while containing avoidable runtime residue that should not be shipped to the operator.
- The project wanted a durable detection and regression rule, not just one-time manual cleanup after a campaign run.
- Reissuing the affected packages in the same pass reduces operator burden and closes the current contamination interval instead of leaving the repair only as future-facing guidance.

Validation notes
- The campaign audit confirmed that only a bounded subset of current helper-skill packages carried runtime residue.
- The updated `dcoir-parity-verifier` and `dcoir-skill-regression-auditor` skill packages pass clean packaging validation.
- Clean reissued packages were built for the affected skills with no `__pycache__/`, `*.pyc`, or `.DS_Store` entries present in the delivered zips.

Next immediate move
- Continue the workflow-state high-risk wave from the live-validated startup, repaired delivery, continuous-execution, and package-cleanliness enforcement baseline while using the updated cleanliness checks for future package verification and readiness claims.
