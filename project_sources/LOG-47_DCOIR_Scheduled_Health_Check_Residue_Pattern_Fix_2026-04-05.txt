DCOIR Scheduled Health Check Residue Pattern Fix - 2026-04-05

Current state id: 2026-04-05-scheduled-health-check-residue-pattern-fix-v1

Purpose
- Record the root cause of the remaining scheduled-health-check failure after the earlier summary and residue-order corrections.
- Record the bounded fix required before the first-wave GitHub automation baseline can be treated as fully settled again.

Observed failure
- The manually triggered scheduled-health-check run still failed in the `Check for packaging residue` step.
- The error was not a real residue hit. The step failed because the `__pycache__` regular-expression string was itself invalid in PowerShell.
- The workflow therefore stopped before Python syntax validation, PowerShell syntax validation, the representative Retrieval suite, and artifact generation.

Root cause
- The residue step used regex-based `-match` expressions with a malformed escaped backslash pattern.
- PowerShell treated the pattern as invalid and threw an error before any meaningful residue classification could happen.

Corrective change
- Replace the regex-based residue matcher with safe non-regex checks:
  - path contains `\__pycache__\`
  - extension equals `.pyc`
  - filename equals `.DS_Store`
- Keep the residue step before Python syntax validation so the workflow still avoids tripping over `.pyc` files generated later in the same run.

Why this matters
- The hygiene rule itself is still useful.
- The failure should reflect actual residue or actual validation problems, not a malformed pattern in the checker.
- One successful rerun after this fix is enough to treat the first-wave baseline as fully proven again.

Best next move
- Apply this fix patch.
- Rerun scheduled-health-check once.
- If it passes, return to the stable-baseline posture and keep the later approved platform-adoption queue as bounded future work.
