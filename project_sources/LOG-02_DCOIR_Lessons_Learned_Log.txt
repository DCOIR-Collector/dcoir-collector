DCOIR Lessons Learned Log

Latest governed lessons added in 3.9.0
- GitHub-primary / Project-bootstrap is cleaner than duplicating readable governed text in both GitHub and Project space.
- Resume reliability improves when the rule is explicit: Project Instructions first, then the GitHub connector against the named repository.
- Native readable script formats in GitHub are preferable once direct repo reads are proven.
- Direct GitHub updates are preferable to patch ZIPs for frequently changing readable text when connector support exists.
- Skills must be updated after a working-source model shift or they will drift behind reality.
- Helper-skill working memory can live safely in GitHub when it stays in a clearly separated human-readable folder and is treated as helper state rather than control-plane authority.
- One canonical markdown memory file per helper skill is easier to inspect and maintain than mixing helper memory into governed project files.
- Converting the highest-value helper skills first is an effective way to prove a new continuity pattern before expanding it across the broader skill set.
- For grouped manual skill replacement, one bundled ZIP is the preferred delivery form because it reduces operator download and update friction.
- The low-level GitHub update path should be validated against the real repository when a new workflow matters; connector-safe commit messages and compact overwrite content reduce avoidable write friction.
- From this project chat, connector-backed GitHub is the primary viable runtime model; direct API auth is not the primary in-chat execution path.
- GitHub persistence is an intended feature for this project, so optimization should focus on batching, branch-safe validation, bounded operations, and fewer fragile retries rather than avoiding writes altogether.
- The connector-backed GitHub operator should expose as much feasible repo, git-object, branch, review, issue, comment, and helper-memory capability as the in-chat GitHub tool surface supports in one release, while leaving unsupported management surfaces deferred.

Durable lessons retained
- Evidence-first, bounded-scope analysis remains the right DCOIR foundation.
- Deep regression should be the default for anything testable before live use and after every patch.
- Good knowledge management has no practical substitute in this project.
