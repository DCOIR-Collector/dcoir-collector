DCOIR Project Transition Checklist

Purpose
- This is the evergreen checklist for applying bounded DCOIR repo or Project-context updates.
- Use it together with the current control plane, the active Airtable plan/task state, and any release instructions included with the current update bundle.
- Do not rewrite this checklist for every normal release.

Current authority baseline
- GitHub repository `malwaredevil/dcoir-collector` is the governed readable source, helper-skill source, control plane, release history, and promoted-decision surface.
- Airtable owns live operational state: queue order, branch priority, active execution order, plans, plan tasks, session carry-forward, operator preferences, validation catalog state, and dedicated helper-skill durable memory where tables exist.
- ChatGPT Project space is a bootstrap/runtime anchor only. Do not recreate a duplicate editable Project-space readable-source repository.
- GitHub todo files and old active-now files are retained as migration/promoted-history surfaces only unless Airtable explicitly reauthorizes them.

Before applying an update
- Confirm current source authority by reading `project_sources/governance/control_plane/CP-01_DCOIR_Version_Manifest.txt` first and `project_sources/governance/control_plane/CP-02_DCOIR_Change_Log.txt` second when current-state claims matter.
- Confirm live execution state from Airtable `Queue Control`, `Work Items`, active `Plans`, and active `Plan Tasks` when branch order or resume state matters.
- Confirm whether the update is one of these classes:
  - affected-file-only GitHub Desktop manual repo-update bundle
  - targeted helper-skill package update
  - grouped helper-skill package update
  - retained supporting-asset refresh
  - special full repo working-tree replacement for T99 final cleanup only
- Confirm that the bundle shape matches the declared update class.

Apply an affected-file-only GitHub Desktop manual repo-update bundle
1. Extract the bundle locally.
2. Confirm the bundle has no wrapper root and no `.git` directory.
3. Confirm every included path is an affected repo-relative path.
4. Copy or overwrite only those paths into the local `malwaredevil/dcoir-collector` working tree.
5. Inspect the GitHub Desktop diff before committing.
6. Commit with the suggested Summary unless the operator intentionally chooses another Summary.
7. Push.
8. Request GitHub readback verification before treating the update as complete.

Apply helper-skill updates
- If exactly one skill is affected, install only that skill ZIP.
- If multiple skills are affected, install the provided affected-skill ZIPs from the outer package.
- After skill installation, verify marker/readback and run the required regression or parity checks before claiming readiness.
- If skill source changed, the governed source under `dcoir_skills/` must also be updated or already match the delivered package.

Apply Project-space/bootstrap updates
- Apply Project Instructions or Project-space bootstrap pointer updates only when the current control plane or operator request explicitly requires it.
- Do not upload every file from `project_sources/` into Project space as a normal update pattern.
- Do not use Project space as a second editable readable-source repository.
- CP-00-style Project-space files are bootstrap pointers only and must not replace GitHub CP-01/CP-02.

Specific checks for this project
- Confirm the current readable collector source remains `project_sources/collector/source/DCOIR_Collector.ps1` unless CP-01 changes it.
- Confirm operator-facing runtime examples use `DCOIR_Collector.ps1` as the emitted collector filename.
- Confirm PowerShell 5.1 remains the local workstation/test lane unless explicitly changed by the control plane.
- Confirm Elastic Defend response-action instructions and local PowerShell workstation instructions are not mixed.
- Confirm `Validation Test Cases` is used as the manual testing catalog for collector/Gemini testing branches.
- Confirm any retained supporting asset or generated ZIP is represented as a supporting artifact, not as editable readable authority.
- Confirm old helper-memory markdown under `dcoir_skill_memory/` is treated as legacy/source-basis history when a dedicated Airtable memory table exists.

After applying the update
- Open a fresh Project chat or continue the current branch only after the operator has pushed the local commit.
- Re-read the affected GitHub files from `malwaredevil/dcoir-collector` and verify the expected content is present.
- Update the relevant Airtable plan task only after readback verifies the remote GitHub state.
- Keep T99 deletion/final cleanup deferred unless the active plan explicitly authorizes T99 work and all cleanup gates have passed.

When to stop and resolve a mismatch
- Stop if the bundle shape does not match the declared update class.
- Stop if an affected-file-only bundle includes unrelated files, a wrapper root, or `.git`.
- Stop if CP-01/CP-02 conflict with Project Instructions or CP-00 on current-state or startup behavior.
- Stop if Airtable live plan/task state conflicts with a proposed branch move.
- Stop if a retained file hard-references a removed path without historical/fallback wording and no bounded fix has been prepared.

Default reminder
- Prefer affected-file-only bundles for bounded repo patches.
- Prefer targeted skill ZIPs for skill changes.
- Reserve full repo working-tree replacement for the special T99 final end-state cleanup lane only.
