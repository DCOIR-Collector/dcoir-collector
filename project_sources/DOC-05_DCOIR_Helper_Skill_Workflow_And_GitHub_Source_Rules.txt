DCOIR Helper Skill Workflow And GitHub Source Rules

Purpose
- This document records the current evergreen workflow rules for DCOIR helper-skill development, post-change regression, GitHub source rollout, closed-loop memory-preflight usage, and session-local write-buffer behavior.

Scope
- These rules apply to project-side DCOIR helper skills and the workflows used to create, update, validate, install, persist, and improve them.
- This document is evergreen workflow guidance, not control-plane authority over the full project.

Core standing rules

1. Closed-loop memory-preflight rule
- Use `dcoir-memory-preflight` before high-friction execution when the task family is likely to have reusable validated procedures, limitations, or failure signatures.
- Use `dcoir-memory-preflight` again after a blocker or failed attempt is successfully overcome whenever the lesson could improve a repeatable workflow, procedure, limitation note, failure signature, or helper-skill/process guidance.
- Do not silently promote every recovered blocker into canonical memory. First classify whether the lesson is one-off or reusable, then stage a bounded promotion candidate.

2. Session-local write-buffer rule
- Relevant DCOIR helper skills may accumulate session-local buffer content during the chat and flush it into GitHub in grouped updates at the next suitable write point instead of writing every small change immediately.
- Session-local buffered content is not cross-session durable unless it is flushed to GitHub or exported into a handoff artifact.
- Preferred flush-check trigger points are:
  - before any GitHub write
  - after blocker resolution
  - when switching major tasks
  - at major milestones
  - before session export or handoff
  - when the operator asks what remains
  - when a helper skill reports meaningful state drift

3. Minimal-operation GitHub write rule
- Prefer staging content in-session and using as few GitHub operations as possible.
- For related existing-file updates, prefer one bounded grouped git-object transaction over one-file-at-a-time churn when the connector lane can do so safely.
- If the grouped lane cannot yet be completed safely in-chat, preserve the intended grouped-write posture and keep the limitation explicit rather than hiding it.

4. Mandatory post-change skill regression rule
- After every helper-skill create or update, pass the result through `dcoir-skill-regression-auditor` before treating the skill as ready for broader use, packaging follow-through, or workflow promotion.
- Packaging success alone is not sufficient proof of runtime correctness.
- Prefer at least one representative success-path test and one failure-gate or edge-case review when feasible.

5. New-skill GitHub-source rollout rule
- When DCOIR helper-skill source is stored in GitHub under the governed one-source-of-truth model, the default new-skill rollout workflow is:
  - build the skill package
  - download the skill zip
  - manually install the skill
  - extract the installed source contents locally
  - add the extracted source to the GitHub working clone through GitHub Desktop
  - then let future readable-source maintenance happen from GitHub

6. Updated-skill GitHub-source rollout rule
- When a DCOIR helper skill already has governed GitHub source, the default update workflow is:
  - build the updated skill package
  - download the updated skill zip
  - manually install the skill update
  - use the GitHub-governed source set as the durable readable source of truth
  - let ChatGPT perform the governed GitHub source update directly when the connector lane can do so safely

Skills most likely to benefit from session-local write buffers
- Highest-value initial candidates:
  - `dcoir-plan-tracker`
  - `dcoir-session-tracker`
  - `dcoir-decision-policy`
  - `dcoir-memory-preflight`
  - `dcoir-skill-regression-auditor`
- Strong secondary candidates:
  - `dcoir-live-test-remediation-planner`
  - `dcoir-validation-orchestrator`
  - `dcoir-change-impact-analyzer`
- Lower or more situational candidates:
  - `dcoir-readme-maintainer`
  - `dcoir-knowledge-doc-maintainer`
  - `dcoir-prompt-pack-assembler`
  - `dcoir-release-scope-builder`
  - `dcoir-promotion-readiness-reviewer`

Staged implementation path

Stage 1
- Govern the workflow rules in readable project sources.
- Preserve the buffer model as session-local only until explicit flush behavior is implemented.

Stage 2
- Upgrade these skills and/or their workflows:
  - `dcoir-memory-preflight`
  - `dcoir-decision-policy`
  - `dcoir-plan-tracker`
  - `dcoir-session-tracker`
  - `dcoir-skill-regression-auditor`
- Add explicit trigger behavior for pre-execution preflight, post-blocker re-check, buffer-state review, and promotion-candidate staging.

Stage 3
- Review the broader `dcoir-*` helper-skill set for whether each skill should support session-local buffering, grouped GitHub flushes, or closed-loop memory-preflight routing.
- Only after that broader review, begin a larger multi-skill source and behavior refresh pass.

Truth rules
- Do not claim cross-session persistence for session-local buffers unless they were flushed to GitHub or exported into a handoff artifact.
- Do not let helper-memory working state override the control plane.
- Do not silently promote reusable lessons into canonical memory.
- Keep the operator informed when grouped-write behavior is desired but not yet fully supported by the current in-chat connector lane.
