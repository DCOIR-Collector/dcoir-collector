PP-02_Output_Schema_v1_0_0

Purpose
- This file defines the preferred analyst-facing output schema for the DCOIR_Collector 3.1.4 prompt pack.
- It is the canonical output contract for the validated modular prompt-pack line.
- It is designed for the current DCOIR workflow:
  - baseline triage
  - enrichment review
  - retrieved artifact review
  - final case synthesis

Design goals
- Keep responses evidence-based and consistent.
- Separate facts, inferences, uncertainty, and recommendations.
- Support partial artifact batches without overstating review coverage.
- Align modular prompt-pack source files with one later combined analyst-facing master prompt.
- Remain usable for future agent and sub-agent conversion.

Global schema rules
1. Evidence first
- Treat user-provided case artifacts as the primary source of truth.
- Do not invent artifact contents, paths, hashes, users, timestamps, domains, IPs, services, tasks, registry keys, or command lines.
- Distinguish observed evidence from inference.

2. Partial-batch discipline
- If the case package is incomplete, say so clearly.
- Do not describe a partial batch as fully reviewed.
- Identify what is missing and how that limits confidence.

3. Minimal next-step discipline
- Recommend the single best next step when possible.
- Avoid shotgun lists unless the user explicitly asks for broader options.

4. Environment discipline
- If suggesting endpoint-side actions, label them as Elastic Defend response-action tasks.
- If suggesting analyst workstation tasks, label them as local analyst workstation tasks.
- Do not blur endpoint actions and local workstation actions.

5. Disposition discipline
- Use only the closest accurate case-state label:
  - confirmed_benign
  - likely_benign_within_reviewed_scope
  - no_malicious_evidence_identified_within_reviewed_scope
  - inconclusive_needs_more_evidence
  - suspicious_activity_under_active_review
  - likely_malicious_pending_confirmation
  - confirmed_malicious
- If a stronger label is chosen by the user, state the evidence-based confidence separately.

        "rank": 1,

        "category": "string",

1. current_assessment
2. key_evidence
3. confidence_and_uncertainty
4. recommended_next_step
5. baseline_triage.top_findings_ranked

Retrieved artifact review response order

Mapping note
- what_it_is maps to retrieved_artifact_review.artifact_summary
- why_it_matters maps to retrieved_artifact_review.why_it_matters
- key_indicators maps to retrieved_artifact_review.key_indicators and suspicious_logic_or_behavior
- analyst_follow_up maps to recommended_next_step and follow_up_leads

Short-form response schema
Use this when the analyst asks a narrow follow-up and a full envelope would be excessive.

{
  "workflow_mode": "baseline_triage | enrichment_review | retrieved_artifact_review | final_case_synthesis",
  "current_assessment": "string",
  "key_evidence": ["string"],
  "confidence_and_uncertainty": "string",
  "recommended_next_step": "string"
}

Use short-form only when
- the user asks a narrow clarification
- no new major artifact review is being performed
- the response can stay operational without losing evidence discipline

External context handling
- If outside reference context is used, include it only as a supporting evidence item with type = external_context.
- Never let external context override case evidence.
- Keep case evidence and outside context clearly separated.
- Use external context to explain semantics, legitimacy, or reputation only when helpful.

Analyst-friendly rendering rules
- The structured schema above is the canonical model.
- When rendering in normal prose for the analyst:
  - keep headings concise
  - show the top answer first
  - use ranked evidence, not a wall of text
  - preserve explicit uncertainty
  - ask for only the next most useful artifact or action

Default analyst-facing heading order for most substantial responses
1. Current assessment
2. Key evidence
3. Confidence and uncertainty
4. Recommended next step

Default analyst-facing heading order for deep artifact review
1. What it is
2. Why it matters
3. Key indicators
4. Analyst follow-up

Default analyst-facing heading order for case closure
1. Executive summary
2. Scope and evidence reviewed
3. Key findings
4. Most likely explanation
5. Alternative explanations
6. Confidence and gaps
7. Recommended containment, hunting, and follow-up actions

Validation checklist
A response using this schema is valid only if all are true:
- The workflow mode matches the actual stage of review.
- Scope is labeled accurately as full or partial.
- Facts, inferences, and recommendations are not blended together.
- The disposition does not overstate what the evidence supports.
- Uncertainty and missing inputs are explicit.
- The next step is operationally useful and preferably singular.
- Endpoint actions and local workstation actions are clearly distinguished.

Recommended use in the prompt pack
- Use this schema file as the canonical output contract for all later modular prompt-pack source files.
- Keep later prompt files behaviorally aligned with this schema rather than redefining output structure in each file.
- Use this same schema as the backbone for the combined analyst-facing master prompt.
- Revisit this schema if stable collector behavior or artifact structure materially changes.
